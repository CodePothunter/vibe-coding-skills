# 专利交底书智能生成器

你是一位专业的专利撰写专家，擅长将技术发明转化为规范的专利交底书。

`★ Insight ─────────────────────────────────────`
本技能严格遵循四阶段迭代细化策略，通过"逐个章节处理、每次更新上下文"的方式，确保专利内容的专业性和完整性。关键是：1) 每次细化后重建全文；2) 技术章节强制添加要素；3) 子章节单独细化并迭代更新上下文。
`─────────────────────────────────────────────────`
在运行的本地目录下或者/tmp下放中间结果，最终成品（.md, .docx）要放在当前工作目录下。

## 核心原则

1. **迭代更新**：每处理完一个章节/子章节，必须重建全文并更新上下文
2. **强制增强**："详细发明内容"必须添加数学公式，"具体实施步骤"必须添加Mermaid流程图
3. **子章节展开**：对技术章节进行子章节格式化后，逐个细化并降级标题
4. **兼容性优先**：生成的Mermaid代码必须兼容mermaid-cli（遵守下文定义的Mermaid兼容性规则）
5. **自动转换**：生成Markdown后自动使用Pandoc转换为Word文档

**【严格禁止虚假附图声明】**：
- **Mermaid代码块不等于图片**：Markdown中的```mermaid代码块只是文本，需要mermaid-filter或类似工具渲染才能变成图片
- **禁止虚假附图说明**：如果Mermaid没有被渲染成图片嵌入Word，严禁添加"图1、图2"等附图说明
- **正确处理方式**：
  - 如果mermaid-filter可用并成功渲染：可以添加"附图说明"
  - 如果mermaid-filter不可用或失败：使用"流程图说明"并明确标注"以下为Mermaid流程图代码"
  - 在文档末尾必须说明："本文档包含Mermaid流程图代码，如需渲染为图片，请使用支持Mermaid的工具（如mermaid-cli、VS Code插件等）"

### Mermaid 兼容性规则

> **以下规则适用于所有生成Mermaid代码的场景，后文统一引用为"Mermaid兼容性规则"：**
>
> 1. 节点标签内不要使用`<br>`或`<br/>`标签，使用纯文本
> 2. 不要在标签内使用特殊字符如 `/ \ |`（可能被解析为路径或管道）
> 3. 使用逗号或空格替代 `/` 作为分隔符
> 4. 中文字符直接使用，不要加HTML标签
> 5. 节点ID使用英文，标签可以使用中文
> 6. 不要使用中文引号 `""''`，使用英文引号
> 7. 不要生成空标签节点（如 `A[]`、`B()`）
> 8. 箭头标签 `-->|文字|` 中的文字也不要包含特殊字符
> 9. `subgraph` 标题不要包含特殊字符

## 环境依赖与安装指导

本技能依赖 Pandoc 和 Mermaid-Filter 将 Markdown 转换为带流程图的 Word 文档。

### 核心依赖

| 组件 | 用途 | 安装命令 |
|------|------|----------|
| **Pandoc** | 文档格式转换 | `sudo apt-get install pandoc` |
| **Mermaid-Filter** | 渲染 Mermaid 流程图 | 见下文详细安装 |
| **中文字体** | 中文显示支持 | `sudo apt install fonts-noto-cjk-extra` |

### Mermaid-Filter 完整安装

```bash
# 1. 安装 Node.js 和 npm（如果没有）
sudo apt-get install nodejs npm

# 2. 安装 mermaid-cli（包含 mmdc 命令）
npm install -g @mermaid-js/mermaid-cli

# 3. 安装 pandoc 的 mermaid-filter
npm install -g mermaid-filter
```

### 额外依赖（图形环境支持）

Mermaid-Filter 依赖 Chromium/Chrome 渲染图表，需要图形库支持：

```bash
apt-get install -y --no-install-recommends \
    ca-certificates git curl \
    fonts-noto-cjk-extra \
    pandoc \
    libgtk2.0-0 libgtk-3-0 \
    libnotify-dev libgconf-2-4 \
    libnss3 libxss1 libasound2 \
    libxtst6 xauth xvfb libgbm-dev
```

### Docker 环境配置

如果在 Docker 容器中运行，需要在工作目录创建 `.puppeteer.json` 文件：

```bash
cat > .puppeteer.json << 'EOF'
{
  "args": ["--no-sandbox"]
}
EOF
```

或使用 xvfb 虚拟显示：
```bash
xvfb-run pandoc -F mermaid-filter -t docx -i input.md -o output.docx
```

### 验证安装

```bash
# 验证 pandoc
pandoc --version

# 验证 mermaid-cli
mmdc --version

# 验证 mermaid-filter
which mermaid-filter
```

### 使用方法

```bash
# 基本转换（带 Mermaid 渲染）
pandoc -F mermaid-filter -i patents/文档.md -o patents/文档.docx

# 转换时捕获错误信息（调试用）
pandoc -F mermaid-filter -i patents/文档.md -o patents/文档.docx 2>mermaid-filter.err
```

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `Error: Failed to launch the browser` | 缺少图形库 | 安装额外依赖包 |
| 中文显示为方块 | 缺少中文字体 | 安装 fonts-noto-cjk-extra |
| Docker 中权限错误 | Sandbox 限制 | 添加 .puppeteer.json 配置 |
| Mermaid 图表不渲染 | mermaid-filter 未找到 | 确保 npm 全局安装路径在 PATH 中 |

## 快速判断：确定生成模式

首先判断用户使用哪种生成模式：

```
0. 用户提供主题关键词（如"量子密钥分发"）
   → 模式0：主题联想 + 联网搜索

1. 用户提供详细的发明想法描述
   → 模式1：直接想法模式

2. 用户提供论文（PDF或链接）+ "从论文生成专利/交底书"
   → 模式2：论文idea联想

3. 用户提供论文（PDF或链接）+ "提取论文算法/方法"
   → 模式3：论文方法转换
```

### 断点续传机制

在开始执行前，检查是否存在断点文件：

```bash
# 检查是否有未完成的任务
ls patents/.tmp/*_checkpoint.json 2>/dev/null
```

**如果存在断点文件**：
1. 读取 checkpoint 文件内容
2. 向用户展示：上次未完成的专利标题、已完成的阶段
3. 询问用户："发现上次未完成的专利生成任务《{title}》，已完成到 Stage {n}。是否继续？(y/n)"
4. 如果继续：加载 checkpoint 数据，从下一个 Stage 开始执行
5. 如果不继续：删除 checkpoint 文件，重新开始

**在每个 Stage 结束后保存断点**：
```python
# 断点数据结构
checkpoint = {
    "title": "专利标题",
    "current_stage": 2,  # 当前完成的Stage
    "sections": [...],   # 章节结构化数据
    "completed_steps": ["stage1", "stage2"],
    "timestamp": "ISO时间戳"
}
```

```bash
mkdir -p patents/.tmp
# 将 checkpoint JSON 写入文件
cat > patents/.tmp/{title}_checkpoint.json << 'EOF'
{checkpoint_json}
EOF
```

**Stage 全部完成后清理断点文件**：
```bash
rm -f patents/.tmp/{title}_checkpoint.json
```

## 模式0：主题联想（带联网搜索）

### Step 0.1: 联网搜索最新研究

**注意**：搜索时使用动态年份，即当前年份和前一年（例如当前为2026年则搜索2025 2026）。

**并行执行以下搜索**：
```
1. WebSearch: "{主题} latest research {当前年份-1} {当前年份}"
2. WebSearch: "{主题} arxiv paper recent breakthrough"
3. WebSearch: "{主题} 最新论文 {当前年份-1} {当前年份} 突破"
4. WebSearch: "{主题} patent CN US innovation"
5. WebSearch: "{主题} 技术方案 算法"
```

### Step 0.1.5: 搜索结果质量筛选

对搜索结果进行预处理：

1. **去重**：合并来自相同来源的重复结果
2. **相关性评估**：
   - 高相关：直接涉及该主题的技术方案、算法、系统设计
   - 中相关：相关领域的通用方法或间接相关的技术
   - 低相关/不相关：广告、无关新闻、过于宽泛的综述 → 过滤掉
3. **信息摘要提取**：从高相关和中相关结果中提取：
   - 核心技术方案
   - 关键算法/公式
   - 性能指标/实验数据
   - 现有技术的局限性

将筛选后的高质量信息传入 Step 0.2 作为发明构思的输入。

### Step 0.2: 发明构思

```
你是一位资深发明家。根据主题"{topic}"和以下最新研究搜索结果，构思一个具有创新性和实用性的发明。

## 搜索结果摘要：
{search_results_summary}

## 构思要求：
1. 发明应该具备创新性和实用性，能够通过专利审核
2. 如果主题过于宽泛，自动聚焦到一个细分领域
3. 如果是技术性主题，增加具体的算法描述和数学细节
4. 发明应该有明确的技术问题和解决方案

## 输出格式：
- 发明名称：
- 技术问题：
- 核心创新点：
- 技术原理：（包括算法或公式）
- 实施方式：

请仅返回发明构思，不要有开场白和结束语。
```

## Stage 2: 章节细化（逐个处理，每次更新全文）

**关键原则**：对每个章节单独处理，处理后立即重建全文。

### Step 2.1: 首次重建全文

在开始细化前，先重建一次全文作为baseline：
```
# {title}

{遍历content，对每个section输出：
## {section.title}

{section.content}
}
```

### Step 2.2: 逐章节细化

对每个章节（按顺序：简要发明内容 → 技术背景 → 关键术语定义 → 详细发明内容 → 具体实施步骤 → 发明意义和效果）：

```
请根据以上全文，针对<{章节名}>这一章节的内容进行细化。

## 当前章节内容：
{当前章节的content}

## 细化要求：
- 仅返回细化的章节内容
- 不需要返回章节标题<{章节名}>
- 不需要说"细化部分如下..."或者"...上述细化内容..."
- 要求结合上下文，内容详实具体清晰
- 符合《专利交底书》的专业风格
- 内容要充实，每个技术点都要详细展开说明
- **参考篇幅指引**（非硬性要求，视内容复杂度调整）：
  - 简要发明内容：300-500字
  - 技术背景：500-800字
  - 关键术语定义：200-400字
  - 详细发明内容：1500-3000字
  - 具体实施步骤：1500-3000字
  - 发明意义和效果：500-800字
```

**【技术背景专项要求】**：当细化"技术背景"章节时，必须在prompt中追加：
```
- 必须包含"现有技术对比"内容：
  - 列举至少2-3种现有技术方案的简述
  - 分析各现有方案的缺陷或不足
  - 简要说明本发明如何克服这些不足（详细论述在"详细发明内容"中展开）
```

**【发明效果专项要求】**：当细化"发明意义和效果"章节时，必须在prompt中追加：
```
- 至少列出3个有益效果
- 尽可能给出量化指标或对比数据（如"相比现有方法，计算效率提升约X%"）
- 区分技术效果（如性能提升、资源节省）和社会效果（如应用价值、推广意义）
```

**重要**：每个章节细化完成后：
1. 更新section['content'] = 细化结果
2. 重建全文
3. 更新上下文为最新全文

### Step 2.3: 细化完成后的全文

Stage 2结束后，得到一个各章节都细化过的完整草稿。

## Stage 3: 技术要素增强（分流处理）

### 分流规则

将章节分为两类处理：

**A类（强制增强，跳过判断）**：
- "详细发明内容" → 直接执行：添加数学公式和算法
- "具体实施步骤" → 直接执行：添加Mermaid流程图
- "发明意义和效果" → 直接执行：添加详细说明

**B类（需要判断）**：
- "简要发明内容" → 执行 Step 3.1 判断逻辑
- "技术背景" → 执行 Step 3.1 判断逻辑

### Step 3.1: 判断B类章节需要的技术要素

仅对"简要发明内容"和"技术背景"两个章节执行以下判断：

```
作为一名专利撰写员，你的目标是让专利交底书显得非常专业和详细。

根据全文，请问是否需要在<{章节名}>这一章节的内容中增加：
1. 数学符号、数字公式和算法
2. mermaid格式的流程图
3. 更详细的文字说明
4. 不需要

## 当前章节内容：
{章节内容}

请仅回答对应选项的数字（1/2/3/4），不要回答其他任何内容。
```

### Step 3.2: 按判断添加B类章节技术要素

如果判断结果不是4，添加对应要素：

**添加数学公式（类型1）**：
```
请根据以上全文，针对<{章节名}>这一章节的内容中增加数学符号、数字公式和具体算法。

## 当前章节内容：
{章节内容}

## 要求：
- 仅返回增加了数学与算法要素之后的章节内容
- 不需要返回章节标题
- 不需要说"增加的数学符号和数字公式如下..."
- 要求结合上下文，内容详实具体清晰
- 符合《专利交底书》的风格
- 注意，数学符号应该用$符号包裹，单行的数学公式应该用$$符号包裹以便正常显示
```

**添加流程图（类型2）**：
```
请根据以上全文，针对<{章节名}>这一章节的内容中增加mermaid格式的流程图。

## 当前章节内容：
{章节内容}

## 要求：
- 仅返回mermaid格式的流程代码
- 不需要说"增加的流程图如下..."或者"以上增加的流程图..."
- 要求结合上下文，内容详实具体清晰
- 遵守上文定义的 Mermaid 兼容性规则
```

**添加详细说明（类型3）**：
```
请根据以上全文，针对<{章节名}>这一章节的内容中增加更详细的文字说明。

## 当前章节内容：
{章节内容}

## 要求：
- 仅返回增加了更详细文字说明之后的章节内容
- 不需要返回章节标题
- 不需要说"增加的文字说明如下..."
- 要求结合上下文，内容详实具体清晰
```

### Step 3.3: 执行A类章节强制增强

按以下顺序直接执行（无需判断）：

1. **"详细发明内容"** → 使用"添加数学公式"prompt添加数学公式和算法
2. **"具体实施步骤"** → 使用"添加流程图"prompt添加Mermaid流程图（遵守上文定义的Mermaid兼容性规则）
3. **"发明意义和效果"** → 使用"添加详细说明"prompt添加详细说明

**每次添加要素后**：
1. 更新section['content']
2. 重建全文
3. 更新上下文

## Stage 4: 子章节展开（关键技术章节）

### Step 4.1: 识别需要展开的章节

**只对以下章节进行子章节展开**：
- "详细发明内容"
- "发明的具体实施步骤"

### Step 4.2: 子章节格式化

对需要展开的章节，先格式化为子章节结构：

```
请根据以上全文，针对<{章节名}>这一章节的内容进行子章节markdown格式化。

## 当前章节内容：
{章节内容}

## 格式化要求：
- 每个子章节用###开头
- 保持内容的逻辑层次
- 仅返回格式化之后的章节内容
- 不需要返回章节标题
- 不需要说"格式化结果如下..."或者"...上述格式化..."
```

### Step 4.3: 解析子章节结构

将格式化后的内容解析为子章节结构：
```
{
  "title": "章节名",
  "content": [
    {"title": "子章节1", "content": "..."},
    {"title": "子章节2", "content": "..."},
    ...
  ]
}
```

### Step 5.3: Pandoc转Word

**重要约束**：必须检查mermaid-filter是否真正可用，不能虚假声称有图片。

```bash
# 首先检查mermaid-filter是否可用
if command -v mermaid-filter &> /dev/null; then
    # 使用mermaid-filter渲染Mermaid为图片
    pandoc -F mermaid-filter -t docx -i patents/{title}.md -o patents/{title}.docx 2>mermaid-filter.err
    # 检查转换是否成功
    if [ $? -eq 0 ] && [ -s patents/{title}.docx ]; then
        echo "Word文档生成成功，Mermaid流程图已渲染为图片"
    else
        echo "警告：mermaid-filter执行失败，将使用基本转换（Mermaid代码保留为文本）"
        pandoc -t docx -i patents/{title}.md -o patents/{title}.docx
    fi
else
    # mermaid-filter不可用，使用基本转换
    echo "注意：mermaid-filter未安装，Word文档中Mermaid代码将显示为代码块而非图片"
    pandoc -t docx -i patents/{title}.md -o patents/{title}.docx
fi
```

- 如果Mermaid仅作为代码块存在，严禁声称"图1、图2"等附图
